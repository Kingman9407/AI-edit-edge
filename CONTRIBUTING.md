# Contributing Guidelines

First off, thank you for considering contributing to this project! It's people like you that make the open-source community such a great place to learn, inspire, and create.

By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md) and our strict non-commercial policy.

## 🚫 Non-Commercial Reminder
As stated in our Code of Conduct, this project is entirely for educational and community use. **By contributing, you agree that your contributions will remain open to the community and may not be used, distributed, or repurposed for commercial monetization by anyone.**

## 💡 How Can I Contribute?

### 1. Reporting Bugs
If you find a bug, please open an issue on GitHub. Before creating a new issue, please search existing issues to see if it has already been reported. When reporting, include:
* A clear and descriptive title.
* Detailed steps to reproduce the issue.
* What you expected to happen versus what actually happened.

### 2. Suggesting Enhancements
Have an idea for a new feature or improvement? We'd love to hear it! Open an issue to discuss your idea before you start writing any code. This helps ensure your idea aligns with the project's goals and saves you time.

### 3. Submitting Pull Requests
Ready to write some code? Awesome! Please follow these steps:
1. **Fork the repository** and create your branch from `main`.
2. **Write clear code:** Try to follow the existing coding style and conventions of the codebase.
3. **Test your changes:** Ensure your code works as expected and doesn't break existing functionality.
4. **Update documentation:** If you change how something works, please update the README or relevant documentation files.
5. **Open a Pull Request (PR):** Write a descriptive PR title and explain what changes you made and why.

## 🧑‍💻 Setting Up for Local Development
1. Clone your fork locally to your machine.
2. Install the necessary dependencies (e.g., `npm install` for frontend/Node apps, or `pip install -r requirements.txt` for Python).
3. Run the development server or test scripts to verify your setup is working.
4. Create a new branch for your feature or bugfix: `git checkout -b feature/your-feature-name`.

Thank you for helping make this project better!
